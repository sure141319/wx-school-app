# Campus Trade 项目上下文与决策边界

> 本文件记录项目的长期产品事实与战略边界。AI 在分析需求、提出方案或修改代码前，应先以本文件为准；不要把本项目按通用大型电商平台进行设计。

## 1. 项目定位

- 项目名称：校园闲置集市（Campus Trade）。
- 当前状态：已经上线运营，并计划长期真实运营。
- 产品形态：面向安徽工业大学学生的校园闲置信息公告栏，而不是提供交易履约保障的电商平台。
- 核心痛点：校园闲置信息散落在多个 QQ 群中，并且很快被聊天消息顶走，导致信息难以持续展示和查找。
- 核心价值：让用户更快找到校内闲置物品。
- 典型使用方式：用户带着明确需求搜索具体商品，或进入分类查找，而不是无目标地浏览推荐流；例如搜索“高等数学教材”。
- 目标用户：安徽工业大学在校生。
- 访问边界：包括非在校生在内的所有人均可浏览商品；“面向校内”属于产品定位和宣传口径，不是由系统身份认证强制保证的准入规则。
- 联系方式边界：未登录访客也可以直接看到和复制卖家公开的 QQ；这是当前明确的产品选择。
- 扩张边界：暂不开放给附近学校，也不按多学校平台设计。
- 预期规模：最多约 2000 名用户，日常并发较低。

## 2. 已确认的产品边界

- 平台负责闲置信息的发布、结构化展示、搜索与筛选。
- 用户发现商品后，通过卖家提供的 QQ 等方式自行联系。
- 聊天、议价、见面、付款和交付均发生在平台之外。
- 平台不介入交易，不提供担保、支付、订单、物流、退款或纠纷调解。
- 不应为了模仿完整电商平台而引入站内交易闭环。
- 即使不介入交易，作为长期运营的信息平台，仍需承担最低限度的内容治理责任，例如违规信息处理、举报、下架和账号封禁。
- 当前商品默认永久展示，直到卖家主动操作；商品数量明显增长后，再考虑设置 30 天或更长的有效期。
- 当前只是简单信息看板，不采集搜索、详情查看、联系方式复制或实际成交等行为数据，无法直接衡量用户是否成功找到商品。
- 当前不记录无结果搜索关键词，也不计划为此增加埋点。
- 商品供给和冷启动暂不作为当前战略讨论重点，除非用户重新提出。
- 搜索排序规则已确认：标题命中优先，其次是描述命中；同级结果再按发布时间从新到旧排序。
- 联系方式公开范围通过平台说明页和“我的”资料编辑提示；发布页不重复展示该提示。

## 3. 技术与系统现状

- `v1/`：Spring Boot 3.3.5、Java 17、MyBatis、MySQL、Redis、MinIO。
- `wxui_v2/`：微信原生小程序。
- `checkui/`：图片审核与商品管理后台。
- 当前采用单体后端，符合项目规模和维护能力，应优先保持。
- 生产系统已经运行，修改时要优先考虑兼容性、数据安全、回滚和运营连续性。
- 当前生产环境没有数据库或 MinIO 的可恢复备份，也没有服务故障告警；这是已知的最高优先级运营风险。
- 生产环境是一台腾讯云轻量应用服务器：Linux、2 核 2 GB、4 Mbps、每月 300 GB 流量。
- MySQL、MinIO、Redis、Nginx 运行在该服务器的 Docker 中；Spring Boot 后端直接运行在宿主机，不在 Docker 中。

### 时间标准

- 项目统一使用北京时间，即 `Asia/Shanghai`（UTC+8），作为所有业务时间、用户可见时间和自然日边界的唯一标准。
- 不得依赖 Linux 宿主机、Docker 容器、JVM、MySQL 会话、浏览器或用户设备的默认时区；后端、数据库连接、接口序列化和前端格式化必须显式遵守北京时间标准。
- 当前接口中的 `LocalDateTime` 一律解释为北京时间墙上时间。前端不得猜测其为 UTC，也不得通过固定加 8 小时掩盖后端或数据库的时区配置错误。
- JWT 有效期、限流窗口、缓存 TTL 等只表示持续时长或绝对时刻的内部逻辑，可以继续使用 `Instant`、UTC 或时间戳；这不等于向用户展示 UTC 时间。

## 4. 产品与技术决策原则

提出功能或架构建议前，依次回答：

1. 它是否直接改善“信息散落、容易被顶走、难以找到”的核心问题？
2. 它是否适合最多约 2000 人、低并发的真实规模？
3. 它是否会让平台被动承担聊天、支付、履约或纠纷责任？
4. 它带来的长期运营和维护成本是否大于用户价值？
5. 是否已有数据、故障或用户反馈证明必须现在解决？

如果以上问题没有明确答案，优先不增加复杂度。

## 5. 优先级指引

### P0：长期运营底线

- 数据库和对象存储备份，并验证恢复流程。
- 基础健康监控、错误日志和关键故障告警。
- 卖家 QQ 对所有访客公开；平台说明页和“我的”资料编辑应明确告知卖家这一点。
- 违规商品举报、管理员下架、必要的账号封禁与操作留痕。
- 明确平台仅提供信息展示的用户协议、隐私说明和风险提示。

### P1：核心价值优化

- 优先提升目标明确型搜索和分类查找的有效性，不投入推荐流或复杂个性化推荐。
- 在商品规模较小时保持分类简单，避免过细分类产生大量空结果。
- 提升商品搜索、分类、校区/地点筛选和排序的有效性。
- 搜索应支持少量受控的校园常用别名，例如“高等数学”与“高数”；优先采用简单关键词归一化，不建设通用中文分词、拼音纠错或模糊搜索系统。
- 教材的版本、出版社、上下册等信息不新增结构化字段，继续由卖家写在标题或描述中。
- 建立清晰的商品生命周期：在售、已售、下架、过期。
- 当前不急于增加自动过期机制；商品数量增长后，再处理长期无人维护的陈旧信息。
- 不建设完整行为埋点或无结果关键词统计；优先通过实际反馈判断搜索是否需要继续优化。

## 6. 已确认的功能需求

### 可配置公告弹窗

- 运营者需要能够随时修改一段文字内容，无需重新发布小程序或重新部署后端。
- 用户每次打开或重新进入小程序前台时检查公告。
- 当前应按“单条可配置公告”设计，不扩展为广告投放、轮播、用户画像、曝光统计或计费系统。
- 公告需要支持启用和停用，停用时不得弹出。
- 公告只包含标题、正文和启用开关；不增加生效时间、失效时间、图片或跳转链接。
- 每日已读状态按设备存储，不跨设备或账号同步。
- 公告“一天一次”的自然日边界按北京时间计算，不跟随设备所在时区变化。
- 弹窗包含“忽略”和“已读”：点击“忽略”不记录已读，下次重新进入前台仍可再次弹出；点击“已读”才记录当天和当前公告版本。
- 同一公告版本在点击“已读”后，当天不再弹出；运营者修改公告后版本递增，即使用户当天已读旧版本也要再次弹出。
- 首次部署默认关闭公告，避免数据库迁移后立即影响线上用户。

### 平台说明与安全提示

- 小程序需要提供集中、公开可访问的平台说明页，未登录访客也能查看。
- 页面必须明确：平台只提供闲置信息发布、展示、搜索与筛选，不提供支付、担保、订单、物流、退款或纠纷处理。
- 页面必须明确：联系、议价、见面、验货、付款和交付均由用户在线下自行完成，平台审核或展示不代表对用户身份、商品真实性或交易安全作出保证。
- 页面必须明确：平台面向安徽工业大学校园场景，但不核验在校生身份，校外人员也可能访问。
- 卖家填写的 QQ、微信号会展示在商品详情中，并对包括未登录访客在内的所有访问者公开；该提示保留在平台说明页和“我的”资料编辑中，发布页不重复展示。
- 安全提示保持具体且克制：建议当面验货、选择公共场所、谨慎提前转账并保留沟通凭证；不扩展为平台担保或纠纷处理承诺。
- 对虚假、违法、侵权或其他违规内容，平台可审核、下架或删除；用户可通过现有反馈渠道联系管理员。

## 7. 当前明确不做

- 微服务、分库分表、消息队列等面向大规模并发的架构。
- 为商品搜索引入 Elasticsearch 等独立搜索系统，除非数据证明 MySQL 已无法满足需求。
- 站内即时聊天、订单、支付、担保交易、物流和售后系统。
- 多学校、多租户或附近高校扩张能力。
- 仅为“架构完整”而增加中间件、抽象层或部署组件。

## 8. 后续战略讨论需要继续确认

下次 AI 对话应优先追问下列问题，而不是直接增加功能：

1. 什么时候说明永久展示已不合适：活跃商品达到某个数量，还是用户开始反馈失效商品过多？
2. 谁负责日常审核、举报和故障处理，能够接受多大的运营工作量？
3. 数据库和 MinIO 的异机备份目标放在哪里，以及希望保留多少天？
4. 故障告警希望发送到邮箱、微信还是其他渠道？

## 9. 给 AI 的协作要求

- 全程使用中文交流，代码标识符保留英文。
- 先确认用户问题和产品边界，再建议技术实现。
- 优先做小而可验证、易回滚的改动。
- 不以“大厂最佳实践”为理由进行超出项目规模的设计。
- 区分三类建议：现在必须做、出现证据后再做、明确不做。
- 发现需求会突破“信息公告栏”边界时，必须先指出新增的运营和责任成本。
